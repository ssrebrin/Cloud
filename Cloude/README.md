# Cloude: Распределенная система вычислений на Java

Cloude — фреймворк для распределенных вычислений, позволяющий выполнять произвольный Java-код (включая лямбда-выражения) на удаленных воркерах. Система автоматически берет на себя вопросы сериализации данных и динамической передачи необходимых скомпилированных классов.

## Архитектура системы

Проект состоит из трех основных компонентов:

1.  **TaskScheduler (Manager)**: Центральный узел, который принимает задачи от клиентов, при необходимости разбивает их и распределяет между свободными воркерами.
2.  **Worker**: Узел-исполнитель. Подключается к менеджеру, получает задачи, динамически подгружает необходимый код и выполняет вычисления.
3.  **Cloud Client (Main)**: Клиентская код, через который пользователь отправляет свои функции на выполнение в кластер.

### Процесс сериализации и передачи кода

Когда вы вызываете метод `cloud.execute(fn, values, extraClasses...)`:

1.  **Сериализация объекта**: Сама функция (объект) сериализуется с помощью библиотеки **Kryo**. Если это лямбда-выражение, система автоматически использует `LambdaWrapper` и стандартный механизм Java (`SerializedLambda`) во избежание проблем с десериализацией.
2.  **Упаковка кода (JAR)**: Класс функции и все дополнительные классы, указанные пользователем (либо перечислением при вызове execute, либо с помощью аннотации @CloudClass), упаковываются в JAR-архив.
3.  **Передача**: Сериализованный объект и JAR-архив кодируются в Base64 (Kryo работает с бинарниками, а Base64 преобразует любые байты в безопасный набор ASCII-символов для передачи в JSON) и отправляются менеджеру в виде JSON-запроса.
4.  **Десериализация и запуск**: Воркер получает данные, использует `CloudClassLoader` для загрузки классов из присланного JAR-архива, после чего Kryo восстанавливает объект функции и запускает его.

## Как запустить

Для корректной работы требуется **Java 21**.

### 1. Запуск менеджера
```bash
./gradlew runScheduler
```
Планировщик запустится на порту `8085`.

### 2. Запуск воркера (в новом терминале)
```bash
./gradlew runCluster
```
Воркер зарегистрируется у менеджера и будет готов к работе.

### 3. Запуск клиента (в новом терминале)
```bash
./gradlew runMain
```

## Ограничения

### Динамическая подгрузка классов
Воркер изначально "не знает" о ваших прикладных классах (например, моделях данных или утилитах). Поэтому при вызове `execute` **обязательно** нужно перечислить все дополнительные классы, которые использует функция, но которых нет в базовом проекте:

```java
// Пример: функция использует классы Calculator и ComplexTask
cloud.execute(f, data, Calculator.class, ComplexTask.class);
```
Если этого не сделать, воркер выбросит `ClassNotFoundException`.

### Поддержка лямбд
Система  поддерживает лямбда-выражения. Однако лямбда, очевидно, должна быть сериализуемой (интерфейс `RemoteFunction` уже наследует `Serializable`).

### Окружение
В текущей версии рекомендуется использовать одинаковую версию Java на всех узлах (рекомендуется Java 21). Версия Gradle 8.8 может выдавать предупреждения на Java выше, чем 21.
